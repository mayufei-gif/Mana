
访问方式：

```text

```

核心使用流程：

```text
把视频同步到 NAS 的 CourseMind/videos
        ↓
CourseMind 后台常驻服务检测文件稳定后扫描入库
        ↓
判断是否像课程视频，非课程进入 ignored 或 needs_review
```

关键原则：扫描可以自动，课程识别可以自动，真实 ASR 不默认无限自动处理。`AUTO_PROCESS_NEW_VIDEO=false` 必须继续作为默认值；后续即使开启自动处理，也必须受课程识别、白名单、每日额度、视频时长、重复调用保护共同约束。

### 3. 任务一：字幕细分与观看体验优化

#### 目标

当前真实 ASR 的 60 秒样片只生成 3 段字幕，其中部分字幕过长，不适合网页播放器观看。需要在 `clean_transcript.json` 到 `smart_subtitle.vtt/srt` 之间增加字幕细分模块。

#### 实现要求

1. 保留 `raw_transcript.json` 原始 ASR 结果。
2. `clean_transcript.json` 继续保存清洗后的原始 ASR 段。
3. 新增 `subtitle_segments` 或 `display_segments`，专门用于播放器显示。
4. `smart_subtitle.vtt` 和 `smart_subtitle.srt` 使用播放器显示字幕段生成。
5. 单条字幕切分优先级：
   - 第一优先级：自然完整的一句话或完整语义句，尽量不要把一句话切成碎片。
   - 第二优先级：观看舒适度，普通字幕建议控制在 6-10 秒内，完整句略长时可以保留。
   - 第三优先级：文本长度，普通字幕建议 25-45 个中文字符，完整句略长时可以放宽到约 50 字。
   - 超长句才切分，优先按中文强标点 `。！？；`，其次按语义连接词 `那么`、`首先`、`第二个`、`第三部分`、`接下来`、`我们来看`。
   - 逗号 `，` 只作为超长句兜底切分点，不能机械地每逢逗号就拆。
6. 如果一段 ASR 文本被切成多段，需要按文本长度比例分配时间戳。
7. 时间戳必须严格递增。
8. `end_time` 必须大于 `start_time`。
9. 最后一条字幕的 `end_time` 不允许超过视频 `duration`。
10. 字幕切分和重建必须能由自动化脚本执行，不能依赖 Codex 手工改字幕。
11. 字幕切分失败时不能影响原始 ASR 文件，必须记录错误阶段。

#### 建议数据结构

```json
{
  "video_id": 1,
  "language": "zh",
  "provider": "aliyun_dashscope",
  "model": "fun-asr-realtime",
  "segments": [
    {
      "segment_index": 0,
      "asr_segment_id": "原始ASR编号，可为空",
      "start": 3.1,
      "end": 19.9,
      "text": "原始识别文本",
      "cleaned_text": "清洗后文本",
      "confidence": null
    }
  ],
  "subtitle_segments": [
    {
      "subtitle_index": 0,
      "source_segment_index": 0,
      "start": 3.1,
      "end": 8.5,
      "text": "适合显示的短字幕"
    }
  ]
}
```

#### 验收标准

1. 60 秒真实 ASR 样片原来 3 条字幕，优化后应明显增加为多条短字幕。
2. 字幕优先保持自然完整的一句话或完整语义句，不允许机械按固定字数切碎。
3. 普通字幕建议控制在 6-10 秒内；如果完整语义句略长，可以优先保留完整句。
4. 每条字幕文本长度适合观看，不出现一整段长文本铺满屏幕。
5. 字幕重建必须能通过自动化脚本完成，不依赖 Codex 手工修字幕。
6. VTT/SRT 正常生成。
7. 播放器点击字幕跳转仍然准确。
8. `/stream` gating 不被破坏：未 ready 返回 409，ready 后返回 200 或 206。

### 5. 任务三：raw_transcript 离线回放机制

#### 目标

避免重复调用百炼，降低测试成本。已经有 `raw_transcript.json` 时，后续调试字幕切分、字幕清洗、章节、重点、笔记时不应再次调用真实 ASR。

#### 实现要求

新增脚本或命令，例如：

```bash
python scripts/rebuild_from_raw_transcript.py --video-id 1
```

该脚本应完成：

1. 读取已有 `raw_transcript.json`。
2. 不调用 dashscope。
3. 重新生成 `clean_transcript.json`。
4. 重新生成 `subtitle_segments`。
5. 重新生成 `smart_subtitle.vtt`。
6. 重新生成 `smart_subtitle.srt`。
7. 重新生成章节、重点、笔记。
8. 输出字幕切分前后对比。

#### 保护要求

1. 默认发现 `raw_transcript.json` 已存在时必须复用。
2. 只有显式传入 `--force-asr` 时才允许重新调用真实 ASR。
3. 日志必须明确显示：

```text
reused raw_transcript
skipped real ASR
regenerated subtitle files
```

#### 验收标准

1. 不配置 `DASHSCOPE_API_KEY` 也能从已有 raw_transcript 重建字幕。
2. 运行离线回放时不会产生新的 ASR 调用。
3. 字幕条数、最大字幕时长、最大字幕字数能输出统计结果。
4. 脚本失败时能显示明确原因。

### 9. 任务七：费用与重复调用保护

#### 目标

避免 NAS 同步大量视频后误触发真实 ASR，导致百炼额度被快速消耗。

#### 强制要求

1. `AUTO_PROCESS_NEW_VIDEO=false` 默认保持关闭。
2. 扫描入库不等于调用 ASR。
3. 只有用户手动点击处理，才允许真实 ASR。
4. 已存在 `raw_transcript.json` 时默认复用。
5. 已存在 chunk 转写结果时默认复用。
6. `ALLOW_REPEAT_ASR=false` 默认关闭重复 ASR。
7. 用户必须显式传入 force 参数才允许重新 ASR。
8. 日志必须记录本次是否调用真实 ASR。
9. 处理前必须展示视频时长和预计处理分钟数。
10. 后续增加每日自动处理分钟数统计。

#### 处理前展示建议

```text
视频时长：58 分钟
ASR 模型：fun-asr-realtime
预计处理音频：58 分钟
AUTO_PROCESS_NEW_VIDEO=false
ALLOW_REPEAT_ASR=false
是否手动开始处理？
```

#### 验收标准

1. 扫描 10 个视频不会自动调用真实 ASR。
2. 只有手动处理一个视频才会入队。
3. 已有 raw_transcript 时重新生成字幕不调用百炼。
4. 需要重复 ASR 时必须显式 force。
5. 日志能查到每次真实 ASR 调用记录。

### 12. 任务九点五：NAS 后台常驻自动化与课程识别

#### 目标

把 CourseMind 从“Codex 辅助开发的项目”推进为“NAS 后台独立运行的学习视频处理服务”。最终用户只需要把视频上传到 NAS 指定文件夹，或让手机相册/微信/网盘等来源同步到 NAS；CourseMind 自动发现稳定文件，判断是否课程，按策略处理，并在手机和电脑网页端呈现为可快速学习的视频。

#### 实现要求

1. 后端服务必须能长期常驻运行，不能依赖 Codex 会话。
2. Docker 容器重启后能恢复扫描、队列、视频状态和历史处理结果。
3. 文件扫描必须先做稳定性检测，避免视频还在同步过程中就开始提取音频。
4. 新视频入库后增加课程识别字段，例如：
   - `content_type`: `course`、`non_course`、`unknown`
   - `course_confidence`: 0-1
   - `auto_process_allowed`: true/false
   - `auto_process_reason`
5. 课程识别初期可以先用轻量规则，不急于接 AI：
   - 文件名和文件夹包含课程、网课、讲义、章节、课时、教程、入门、考试、会计、PLC、数学等关键词。
   - 视频时长超过最小时长阈值。
   - 文件位于课程白名单目录。
   - 非课程黑名单目录不自动处理，例如相册、生活、聊天临时文件、下载缓存。
6. 识别为课程后也不能无条件调用真实 ASR，必须继续检查：
   - 是否已有 `raw_transcript.json`
   - 是否已经处理完成
   - 是否超过单视频最大时长
   - 是否超过每日/每月 ASR 分钟数
   - 是否处于白名单目录
   - 是否启用 `AUTO_PROCESS_NEW_VIDEO`
7. 未通过自动处理条件的视频应进入 `needs_review`，让用户网页端手动确认。
8. 非课程视频应进入 `ignored` 或保持 `pending`，不能占用真实 ASR 额度。
9. 自动处理日志必须记录每个决策原因，便于排查为什么处理或为什么没处理。
10. 手机和电脑访问同一个网页学习入口，ready 视频都应支持字幕同步、字幕跳转、章节跳转、重点跳转、播放进度恢复。

#### 建议配置

```env
AUTO_PROCESS_NEW_VIDEO=false
AUTO_CLASSIFY_NEW_VIDEO=true
AUTO_PROCESS_COURSE_ONLY=true
AUTO_PROCESS_WHITELIST_DIRS=/videos/courses
AUTO_PROCESS_BLACKLIST_DIRS=/videos/photos,/videos/chat_tmp
MIN_COURSE_VIDEO_SECONDS=180
MAX_AUTO_ASR_VIDEO_SECONDS=3600
DAILY_ASR_LIMIT_SECONDS=3600
FILE_STABLE_SECONDS=120
```

#### 验收标准

1. NAS Docker 后台服务不依赖 Codex，也能持续扫描新视频。
2. 正在同步中的大视频不会被提前处理。
3. 放入课程目录的视频能被识别为 `course`。
4. 放入相册/聊天临时目录的视频不会自动调用真实 ASR。
5. 默认 `AUTO_PROCESS_NEW_VIDEO=false` 时，只入库和识别，不自动烧额度。
6. 显式开启自动处理后，也只处理符合课程识别、白名单、额度和时长限制的视频。
7. NAS 重启或 Docker 重启后，已完成视频仍可播放，未完成任务能继续或明确失败。
8. 手机浏览器和电脑浏览器都能打开网页端学习 ready 视频。
